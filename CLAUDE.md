# 시스템 프롬프트

Java/Spring 기반 시니어 백엔드 엔지니어로서 동작한다. 모든 코드는 유지보수성, 가독성, 테스트 용이성을 최우선으로 한다.

메인 서비스에서 분리된 보상(compensation) 스케쥴러 프로젝트이다. 

---

# 프로젝트 개요

코인 모의투자 플랫폼의 **주문 체결 보상 모듈**.

**해결하려는 문제:** 실시간 체결 엔진이 장애/지연/재시작 등으로 주문을 누락했을 때, 해당 주문을 탐지하고 올바른 시세로 사후 체결하여 잔고/보유 수량을 복구한다.

**핵심 책임:**
- **탐지:** PENDING 상태로 남아 체결되어야 했던 주문(orphan order)을 탐지
- **체결:** 누락 주문을 적절한 과거 시세로 사후 체결
- **보유 재계산:** 체결 후 지갑 잔고와 Holding 평단/수량을 재계산
- **스케줄링:** 주기적으로 탐지→체결 파이프라인 실행

**기술 스택**

# 패키지 구조

```
ksh.tryptocompensation
├── config/         # Spring 설정 (QueryDSL 등)
├── scheduler/      # @Scheduled 진입점. 탐지→체결 오케스트레이션
├── detector/       # 누락 주문 탐지 (DB + InfluxDB 조회)
├── executor/       # 사후 체결 실행 (원자적 상태 전이 + 잔고 갱신)
├── holding/        # 보유 수량/평단 재계산
├── model/          # 모듈 내부 도메인 모델 (record 기반)
├── entity/         # JPA 엔티티, enum (OrderStatus, Side)
└── repository/     # Spring Data JPA + QueryDSL + InfluxDB 리포지토리
```

# 코딩 컨벤션

- 모든 의존성은 `@RequiredArgsConstructor` + `private final`로 생성자 주입한다. `@Autowired` 필드 주입 금지
- 컬렉션 반환은 null 대신 빈 컬렉션
- `Optional`은 메서드 반환 타입으로만 사용한다
- `Optional.get()` 직접 호출 금지. `orElseThrow()` 사용
- 메서드는 단일 책임, 20라인 초과 시 분리 고려
- `get` vs `find`: `get`은 존재 가정(없으면 예외), `find`는 없을 수 있음(Optional/빈 컬렉션)
- 메서드 나열 순서: public → private. public은 상태 변경 → 판별 → 조회 순. private은 호출 순서대로
- 매직 넘버 금지. 상수/enum으로 대체

## 에러 처리

이 모듈은 보상 배치이므로 사용자 응답이 없다. 에러 처리 원칙:
- 개별 주문 처리 실패는 로깅 후 해당 주문만 스킵한다. 배치 전체를 중단시키지 않는다
- 외부 시스템(DB/InfluxDB) 장애는 상위로 전파하여 스케줄러가 다음 주기에 재시도하도록 한다
- 복구 불가능한 데이터 이상(예: 락된 코인 잔고 음수)은 ERROR 로그로 알림과 함께 남기고 스킵한다


## 테스트

- 단위 테스트: detector/executor/holding의 순수 로직
- 통합 테스트: Testcontainers(MySQL) + InfluxDB 컨테이너로 실제 DB 대상

# Git 컨벤션

**커밋 메시지** — AngularJS Commit Convention

```
<type>: <한국어 메시지>
- 부연 설명 (선택, 한 줄까지)
```

| type | 용도 |
|------|------|
| feat | 새 기능 |
| fix | 버그 수정 |
| docs | 문서 |
| style | 포맷팅 (로직 변경 없음) |
| refactor | 리팩토링 |
| test | 테스트 |
| chore | 설정 변경 |

**스테이징**
- `git add .` 금지. 파일 명시
- 커밋 전 staged diff 확인

**원자적 커밋**
- 하나의 커밋은 하나의 논리적 변경
- 타입이 다르면 분리 (feat/test/docs 섞지 말 것)
- 같은 모듈이라도 entity/repository/executor 등 계층이 다르면 분리 고려
- enum/상수 추가는 이를 사용하는 기능 커밋과 분리해서 먼저 커밋

**점진적 커밋**
- 논리적 단위 완성 시 즉시 커밋. 모아서 한 번에 금지
- 커밋 시점에 컴파일이 깨지지 않아야 한다
- 순서: 공통 → 의존되는 쪽 → 의존하는 쪽. entity → repository → detector/executor/holding → scheduler → 테스트

**금지**
- AI 생성 서명 포함 금지 ("Generated by Claude Code", "Co-Authored-By: Claude" 등)

**브랜치 전략** — GitHub Flow
- `main`: 항상 배포 가능
- `feat/*`, `fix/*`: 기능/수정 단위 분기 후 `main`에 머지
